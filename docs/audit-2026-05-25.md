# HydroMemory — Project Audit

**Date:** 2026-05-25
**Scope:** full repository — v1 core, v2 §9 OS/platform layer, v2.1 real-model backends.
**Method:** six parallel review lanes — (1) core lifecycle, (2) storage/HQL/intelligence/surfaces, (3) vault + governance security, (4) bus + platform security, (5) test-suite quality, (6) docs/ADR/PRD-traceability/TS-parity — plus a full test run and `ruff`/`mypy`.

> **Verdict:** a genuinely well-engineered reference implementation. The v1 core is correct and clean; tests and docs are unusually strong. Risk is concentrated in the newest layer — the v2 platform/bus access-control *default wiring* — where several paths fail **open** in ways that contradict their own ADR guarantees. None is catastrophic under a local single-user threat model, but several gaps are wider than the docs claim.

## Composed risk — the platform layer fails open by default

The three platform findings **compose**. A non-owner app on the **default** `register_app` wiring can defeat the L4 grant boundary that is supposed to isolate it:

1. The default bus is **repo-less**, so its permission gate fails **open** (H1) → the app's subscribers receive every event (droplet id, reservoir, phase, source) regardless of READ permission.
2. `AppMemory.recall` honors a **caller-supplied** `is_user_proxy` flag to skip grant enforcement (H2) → a forged user-proxy identity reduces L4 to bare `check_access`.
3. The app's vault falls back to the **unscoped** raw engine repo (M2) → there is no structural scope pre-filter, so isolation rests entirely on the (now-bypassed) grant check.

Worst case: a compromised app reads cross-app droplet **content** and observes all memory-bus activity. This is the single most important outcome of the audit; the v2 platform layer (L1–L4) is the newest code and its *defaults* are where the gaps cluster.

**Cross-cutting theme:** identity/trust is *self-asserted and unverified* (server trust strings; `is_user_proxy`). For a local single-user reference impl that is a defensible threat model — but it is the layer where "documented limitation" and "real defect" blur, and it is not written down as an explicit threat-model boundary.

## What's solid (balance)

Encryption field-selection matches the docs **exactly** (`__vault__` token; only `purity` survives in plaintext state; `__vault__` stripped on read). `check_access` is on every vault data path; rotation and keyless→encrypted migration are owner-gated, idempotent, and fail-loud; `NullCipher` cannot silently substitute when a key is set. The grant narrow-never-widen AND is correct; the mesh has a working cascade depth guard. The vector index handles cosine/zero-vector/empty edges correctly. TS↔Python enum & endpoint parity has **zero drift** (`test_ts_parity.py` pins it). The acceptance tests (PRD §12 A–F, §16 metrics) assert real engine-produced state, not smoke. The uncommitted `docs/vault.md` edit is accurate (both new CLI subcommands exist) and safe to commit.

**Notes on residual scope:** H4 settles a transient resting phase to GROUNDWATER (from RIVER) or LIQUID before persistence. M1's head watermark defeats *naive* tail-truncation, but it lives in the same DB as the log — a full-DB-write attacker who also forges the watermark remains out of scope; true tamper-evidence needs an external/signed anchor (the docstring now says so honestly).
